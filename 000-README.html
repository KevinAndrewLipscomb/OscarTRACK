000-README


Abstract

This document describes additional steps that you must perform, after checking out the OscarDotNet project from its repository, to achieve a clean compilation or to execute the application. This document does not cover how to meet other prerequisites, such as setting up an ASP.NET server, or creating and populating the database schema.

1 Recommended methods

The use of NTFS reparse points (junction points and hardlinks) is recommended in this document because it allows Subversion operations to take place without undesirable impact to project environments, and without violating Subversion rules. If you simply exported OscarDotNet from its repository and you do not intend to perform subsequent updates or any other Subversion operations on it (not recommended), you can simply make copies instead of NTFS reparse points.

2 Establish hooks to adjunct projects

2.1 Hook up the KiUsercontrolFoundry project

OscarDotNet uses UserControls from the KiUsercontrolFoundry adjunct project. The KiUsercontrolFoundry is used by multiple production-eligible projects and is version controlled independently from those production-eligible projects. To hook up the KiUsercontrolFoundry project, do the following:

  1. Check out the https://kveo-it-project.svn.sourceforge.net/svnroot/kveo-it-project/trunk/foundry/usercontrol/ki project. Put it in a KiUsercontrolFoundry folder. Make sure the KiUsercontrolFoundry folder is not inside the OscarDotNet folder tree.
  2. Create a junction point named ki inside the OscarDotNet/usercontrol folder that points to the KiUsercontrolFoundry/usercontrol/ki folder. You should end up with two folders inside the OscarDotNet/usercontrol folder: app and ki. You can make junction points with Michael Elsdoerfer’s NTFS Link tool or, on Vista-based systems (including Windows 7, Windows Server 2008, and later systems), the prototype commands would be:

    cd OscarDotNet/usercontrol
    mklink /j ki KiUserControlFoundry/usercontrol/ki

2.2 Hook up the KiDotnetFoundation project (only if compiling)

OscarDotNet uses object code from the KiDotnetFoundation adjunct project. The KiDotnetFoundation is used by multiple production-eligible projects and its source code is version controlled independently from those production-eligible projects. The object code is already provided as OscarDotNet/bin/ki.dll to enable execution of OscarDotNet in a production environment, but to hook up the KiDotnetFoundation project for the sake of a clean compile of OscarDotNet, do the following:

  1. Check out the https://kveo-it-project.svn.sourceforge.net/svnroot/kveo-it-project/trunk/foundation/dotnet/ki project. Put it in a KiDotnetFoundation folder. Make sure the KiDotnetFoundation folder is not inside the OscarDotNet folder tree.
  2. Create a junction point named ki inside the OscarDotNet/lib folder that points to the KiDotnetFoundation folder. You should end up with a folder inside the OscarDotNet/lib folder called ki. You can make junction points with Michael Elsdoerfer’s NTFS Link tool or, on Vista-based systems (including Windows 7, Windows Server 2008, and later systems), the prototype commands would be:

    cd OscarDotNet/lib
    mklink /j ki KiDotnetFoundation

3 Establish hooks to appropriate config files

OscarDotNet will not run without Web.config and unpublished.config files in its top-level folder. Because the contents of those files are environment-specific, and we wish to keep the canonical ASP.NET parts of the OscarDotNet folder tree environment-neutral, those config files are kept in a non-ASP.NET-canonical OscarDotNet/config subfolder tree. To hook the config files that are appropriate for your environment into the top-level OscarDotNet folder, do the following:

  1. If there is no unpublished.config file inside the OscarDotNet/config folder tree (this should be true for projects kept in public repositories), copy the OscarDotNet/config/template-unpublished.config-template file to an appropriate subfolder of OscarDotNet/config, rename it to unpublished.config, and revise its contents appropriately. Do not add any unpublished.config file to any public repository.
  2. Create a hardlink in the OscarDotNet top-level folder that points to the appropriate unpublished.config file inside the OscarDotNet/config folder tree.
  3. Create a hardlink in the OscarDotNet top-level folder that points to the appropriate Web.config file inside the OscarDotNet/config folder tree.

You can make hardlinks with Michael Elsdoerfer’s NTFS Link tool or, from the Windows command line, the prototype commands would be:

cd OscarDotNet
fsutil hardlink create unpublished.config config/{path-to-appropriate-unpublished.config-file}
fsutil hardlink create Web.config config/{path-to-appropriate-Web.config-file}
